# Before Build Week: the honest starting line

Snapshot boundary: immediately before 2026-07-13.

## What existed

The creator had practical experience tutoring mathematics, producing
educational content, applying expert corrections, and using deterministic
release checks. That experience supplied the problem insight:

> A valuable expert correction can fix one artifact without proving that every
> dependent artifact is now consistent.

General engineering practices also existed before the event: preserve source
evidence, distinguish uncertainty from fact, validate important transitions,
and require explicit approval for consequential changes.

These were experience and principles, not a LessonProof product or codebase.

## What did not exist

Before the competition window there was no:

- product named LessonProof;
- correction-to-release application, repository, interface, or API;
- synthetic lesson and correction dataset for public demonstration;
- evidence-bound GPT-5.6 repair-plan contract;
- exact-quote resolver, editable-path allowlist, or dependency-closure check;
- human approval and deterministic verified-release state machine;
- release proof hash, guarded undo journal, or per-browser session boundary;
- LessonProof test suite, deployment path, documentation, or submission media.

## Clean-room boundary

The competition implementation began as a standalone repository. It does not
import prior source code, personal records, private educational material,
credentials, production configuration, screenshots, media, or databases.

The only inherited inputs are the creator's problem knowledge and general
engineering judgment. All public fixtures are synthetic and original.

## Qualifying thesis

The judged work is a new, narrow Education product:

> Capture an expert correction, bind it to inspected evidence, use GPT-5.6 to
> propose a bounded repair, and issue release proof only after human approval
> and deterministic verification.
